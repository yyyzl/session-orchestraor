# 会话编排器脑暴整理（初稿）

日期：2026-02-28  
状态：脑暴收敛中（未冻结）

## 1. 目标与背景

你计划扩展现有的任务执行流程，核心目标是：

1. 支持长程任务分段执行，避免单窗口上下文过长导致质量下降。
2. 按任务类型决定是否清理上下文并新开窗口。
3. 将“每个上下文窗口必须执行的固定步骤”流程化、不可遗漏。
4. 支持提示词与执行编排的可配置化（必选步骤 + 可选步骤）。
5. 全程有记录文件，能追踪每轮执行进度与结果。
6. 提供人工介入界面：除 AI 驱动外，人工可随时向 Codex 发送消息。

## 2. 你当前明确提出的功能点

### 2.1 长程任务分轮执行

1. 一个任务文件可能有 10 个子任务。
2. 第一轮可能只完成其中 2 个。
3. 系统要能自动解析“已完成/未完成”状态。
4. 然后进入下一轮：清理上下文或新开窗口，继续剩余任务。

### 2.2 会话窗口策略（按任务类型）

1. 开发类任务：更倾向每次阶段完成后新开窗口执行下一段。
2. 规划类任务：不一定清除上下文，可能保持连续讨论更高效。
3. 第一版本即要求：由动态策略引擎决定“保留上下文 or 新开窗口”。
4. 你已确认：`dev` 的连续未完成阈值 `N=1`（一旦未完成即触发切窗条件）。
5. 你已确认：`dev` 在单窗口内完成固定步骤后，必须开启新窗口进入下一阶段。

### 2.3 记录与恢复

1. 每次执行结果都要落盘记录（进度文件）。
2. 新窗口启动时读取记录，恢复到正确阶段继续执行。
3. 记录应至少包含：任务 ID、当前阶段、已完成步骤、待办步骤、最近产物摘要。
4. 对 `dev` 任务，切窗记录不能只写摘要，必须附带任务目标与上一轮 Git 提交信息（见 4.7）。

### 2.4 可配置提示词与流程编排

你希望提示词可自定义，并定义每个上下文窗口必须执行的步骤模板。  
此外，还要支持“哪些步骤可选，哪些必做”的编排规则。
你已补充：`git提交` 的提示词要集中在一个地方统一设置。
你已确认：`planning` 场景遇到无代码变更时，`git提交` 采用强制空提交。
你已确认：`dev` 场景遇到无代码变更时，禁止空提交，`git提交` 步骤判定失败并触发修正流程。
你已确认：`git提交` 粒度固定为“每轮（每个成功窗口）一次”。
你已确认：进度解析统一基于结构化输出，不采用自然语言抽取。
你已确认：必做步骤中的“实现”不再细分，保持单步骤语义。
你已确认：失败重试策略为“重试当前步骤”，不回退到上一步。
你已确认：支持运行时切换模型（按任务类型、步骤或人工指令触发）。
你已确认：每轮模型默认由策略自动决策（可人工覆盖）。
你已确认：支持调整思考难度（推理强度分级，可配置覆盖）。
你已确认：人工消息优先级为最高，采用“立即打断并推送新消息”策略。

## 3. 固定不可变流程（你已明确：不可遗漏）

以下步骤是每个会话窗口的硬性流程：

1. `$start`：单独执行这个命令
2. `$before-frontend-dev` 或 `$before-backend-dev`：根据你开发的是前端还是后端，选择对应的命令，或者两个一起使用
3. 说需求（此时 AI 会帮你实现）
4. `$check-frontend` 或 `$check-backend`：根据你开发的是前端还是后端，选择对应的命令，或者两个一起使用
5. `$finish-work`：进行收尾工作
6. git提交
7. `$record-session`：录制本次会话

说明：

1. 上述步骤是强约束，不允许缺失。
2. 前后端步骤是非二选一分支，但每次必须落到其中一个分支。
3. 可选动作只能插入，不得替代以上必做动作。
4. 在 `dev` 任务中，当以上步骤在当前窗口全部完成后，策略必须输出 `start_new_window`。

## 4. 你想要的“编排器”能力（方向草图）

### 4.1 流程规则层

1. 定义必做步骤（required steps）。
2. 定义可选步骤（optional steps）。
3. 定义步骤顺序与前置依赖。
4. 执行后自动校验是否漏步。

### 4.5 提示词配置中心（V1 建议纳入）

1. 所有步骤提示词集中管理（尤其 `git提交` 提示词）。
2. 支持模板变量：`{task_id}`、`{window_id}`、`{stage}`、`{changed_files}`、`{summary}`。
3. 支持按任务类型覆盖（`dev` / `planning`）。
4. 执行时从同一配置源读取，避免多处散落导致不一致。
5. 变更配置后，新窗口自动生效（无需改代码逻辑）。
6. `planning` 场景在无变更时，必须命中空提交模板（`git commit --allow-empty`）。
7. `dev` 场景在无变更时，必须命中“禁止空提交”策略（返回失败并要求补充有效变更）。

### 4.6 人工介入与实时消息通道（V1 必做）

1. 提供一个可视化界面，显示当前任务、当前窗口、当前轮次、当前步骤状态。
2. AI 自动执行期间，人工可随时发送消息给 Codex（实时插入当前会话）。
3. 人工消息必须写入结构化日志，记录 `operator_id`、发送时间、消息内容、影响范围。
4. 人工消息默认不绕过固定步骤链；如需跳步，必须走显式 `manual_override` 标记并留痕。
5. 人工消息可触发三类动作：
   - 补充约束（继续当前流程）
   - 纠偏指令（重试当前步骤）
   - 策略干预（强制切窗或强制保留）
6. 所有人工介入都要体现在每轮结果中，便于复盘与审计。
7. 人工消息优先级固定为最高：到达后立即打断当前自动流程，并立刻向 Codex 推送一条新消息。
8. 打断后流程语义：
   - 当前自动步骤置为 `interrupted` 并记录 `operator_interrupt` 事件；
   - 先处理人工消息对应的 turn；
   - 处理完成后再依据策略决定恢复当前步骤或进入下一步骤/切窗。
9. 人工打断行为不得绕过审计：必须记录打断时间、被打断步骤、恢复动作。
10. 人工消息发送规则：不区分“运行中”或“未运行”，均允许直接发送并即时生效。
11. 人工消息为高优先级插入，不改变固定流程定义；原流程在人工指令处理后按状态机继续。

### 4.2 上下文窗口层

1. 支持“同窗口继续”与“新窗口继续”两种模式。
2. 每轮结束后根据规则判断是否切窗。
3. 切窗时自动注入下一轮提示词（含任务目标 + 进度摘要 + 上一轮提交信息 + 待办）。
4. 新窗口语义明确为：`/new = 新 thread（同进程，默认模式）`。
5. 对 `dev` 任务，`start_new_window` 前必须通过 `Context Handoff` 校验；校验失败不得切窗。

### 4.3 任务进度层

1. 自动解析当前轮完成了哪些子任务。
2. 更新任务状态文件。
3. 生成下一轮执行指令。

### 4.7 Context Handoff（V1 必做，尤其 dev）

切窗时必须生成结构化 Handoff 包，并自动注入新窗口首条提示词。  
对 `dev` 任务，下列字段为强制项：

1. 任务信息：
   - `task_id`
   - `task_goal`（任务是什么、验收标准是什么）
   - `current_stage`
2. 进度信息：
   - `completed_steps`
   - `pending_steps`
   - `last_round_result`
3. 代码工作区状态：
   - `last_commit_id`（上一轮 Git commit id）
   - `last_commit_message`（上一轮提交内容/提交说明）
   - `last_commit_scope`（可选：本次提交涉及的模块或目录）
4. 下一步信息：
   - `known_issues`
   - `next_actions`

Handoff 校验规则（dev）：

1. 缺少 `task_goal`、`last_commit_id`、`last_commit_message` 任一项，禁止切窗。
2. 新窗口首条提示词必须包含三段：任务是什么、上一轮提交信息、下一步动作。

### 4.8 模型切换与思考难度控制（V1 必做）

1. 模型控制：
   - 支持默认模型配置（`default_model`）。
   - 支持按任务类型路由模型（如 `dev`、`planning`）。
   - 支持按步骤覆盖模型（如“实现”步骤可切到编码更强模型）。
   - 支持每轮自动选模：策略引擎根据任务类型、步骤状态、历史质量自动决定本轮 `model_id`。
   - 支持人工消息触发即时切换模型。
2. 思考难度控制：
   - 提供标准档位：`quick` / `balanced` / `deep`。
   - 支持按任务类型和步骤覆盖默认档位。
   - 人工可随时发指令调整档位（例如“本轮切 deep”）。
3. 生效规则：
   - 常规配置变更从下一 turn 生效。
   - 人工即时切换可打断后立即生效。
4. 审计要求：
   - 每轮回执必须记录 `model_id` 与 `reasoning_level`。
   - 模型切换和难度切换必须记录触发来源（策略/人工）。

### 4.4 动态策略层（V1 必做）

1. 输入信号：
   - 任务类型（dev / planning）
   - 当前轮完成率（例如 2/10）
   - 最近输出质量（是否反复追问、是否跑题、是否未达验收）
   - 上下文负载（轮次数、摘要长度、历史错误次数）
2. 决策输出：
   - `continue_same_window`
   - `start_new_window`
3. 默认策略建议：
   - `dev`：强偏向切新窗口（质量优先，且完成固定步骤后强制切窗）
   - `planning`：偏向保留窗口（连续推理优先）
4. 强制切窗条件（任一命中）：
   - `dev` 连续 N 轮未完成阶段目标（当前设定 N=1）
   - 连续出现“仅追问不产出”
   - 上下文超过阈值（例如轮次或摘要长度超限）
   - `dev` 当前窗口已完整执行固定步骤链（`$start` 到 `$record-session`）
5. 强制保留条件（任一命中）：
   - 正在收敛一个规划结论，且质量稳定
   - 本轮目标依赖上一轮细粒度上下文，重建成本过高
6. 所有策略判断结果必须记录到进度文件，便于审计和调优。
7. 修正/重试规则：
   - 任一步骤失败后，先执行修正动作，再重试该步骤。
   - 单步骤最大重试次数默认 `1`（失败后仅重试一次）。
   - 超过重试上限后，策略必须输出 `start_new_window`。

## 5. 建议的最小可落地版本（MVP）

第一阶段先实现最小闭环：

1. 支持任务类型开关：`dev` / `planning`。
2. 支持固定必做步骤校验（漏任一步骤即失败）。
3. 支持进度文件读写（JSON），并以结构化输出驱动进度解析。
4. 支持长任务分段与下一轮续跑。
5. 支持动态切窗策略引擎（V1 必含），并提供可配置阈值。
6. 每轮输出必须包含“策略判定依据 + 判定结果 + 执行动作”。
7. 默认参数落地：`dev.unfinished_threshold_n = 1`。
8. 默认动作落地：`dev` 完成固定步骤链后强制切窗。
9. 新窗口默认实现：`/new => 新 thread（同进程）`。
10. 提供统一提示词配置入口（至少包含 `git提交` 模板）。
11. `git提交` 步骤执行时，必须读取配置中心模板后再生成提交指令。
12. `git提交` 粒度固定：每轮（每个成功窗口）执行一次。
13. `planning` 无代码变更时，必须执行空提交并记录到 session。
14. `dev` 无代码变更时，禁止空提交；当前窗口标记为未完成并进入修正或重试。
15. 默认重试参数：`step.max_retry = 1`；超限后强制切窗。
16. 提供人工发送消息入口（UI 或控制台面板），可在自动执行中即时介入。
17. 人工介入事件必须进入统一回执与指标体系（可追溯、可统计）。
18. `dev` 切窗必须携带完整 Context Handoff（任务目标 + 上一轮提交信息）。
19. Handoff 缺关键字段时，阻断切窗并触发修正流程。
20. 人工消息默认行为为“立即打断 + 新消息推送”，不采用排队模式。
21. 支持运行时模型切换（任务级/步骤级/人工指令）。
22. 支持思考难度分级（`quick` / `balanced` / `deep`）及覆盖策略。
23. 每轮回执必须记录 `model_id`、`reasoning_level` 及切换来源。

## 10. 已确认的 V1 技术规范补充

### 10.1 状态机码表固定（不可漂移）

步骤状态只允许以下枚举值：

1. `pending`
2. `running`
3. `passed`
4. `failed`
5. `retrying`
6. `skipped`
7. `interrupted`（被人工消息打断）

说明：

1. 禁止新增同义状态（例如 `done`、`ok`、`error`）。
2. 对外日志、回执、统计统一使用这套码表。

### 10.2 结构化回执格式固定（每步一条）

每个步骤执行后都必须输出统一 JSON 回执，用于恢复、审计和指标统计。

```json
{
  "run_id": "run_20260228_001",
  "window_id": "window_03",
  "round_index_in_window": 2,
  "handoff_id": "handoff_window03_to_window04",
  "step_id": "step_git_commit",
  "step_name": "git提交",
  "status": "passed",
  "initiated_by": "ai",
  "operator_event_id": null,
  "model_id": "gpt-5-codex",
  "reasoning_level": "balanced",
  "switch_source": "policy",
  "evidence": {
    "last_commit_id": "abc1234",
    "last_commit_message": "feat(auth): 完成登录态校验与错误提示优化"
  },
  "duration_ms": 12840,
  "next_step": "$record-session",
  "timestamp": "2026-02-28T12:34:56Z"
}
```

### 10.3 幂等键规范（防重复执行）

1. 每个窗口有唯一 `run_id`。
2. 每个步骤有唯一 `step_id`。
3. 执行键固定为 `run_id + step_id`。
4. 重试时沿用同一幂等键并递增 `attempt`，禁止产生重复提交、重复记录。

### 10.4 进度文件写入安全（文件锁 + 原子替换）

1. 写入前先加文件锁，防并发写冲突。
2. 先写临时文件（如 `progress.json.tmp`）。
3. 写入成功后原子替换正式文件（rename/replace）。
4. 任何中断都不得破坏正式进度文件可读性。

### 10.5 恢复策略固定

1. 异常重启后从“最后一个未完成步骤”继续执行。
2. 已 `passed` 的步骤不回滚、不重复执行。
3. 进入恢复流程时必须写一条恢复事件日志（包含恢复起点和原因）。

### 10.6 可观测性指标（V1 最小集）

至少记录以下指标：

1. 成功率（按步骤、按窗口、按任务）。
2. 重试率（含失败后重试次数分布）。
3. 平均每窗口耗时。
4. 切窗原因分布（策略触发类型）。
5. 任务每轮结果（通过/失败/重试中）。
6. 任务每轮耗时。
7. 当前窗口的轮次索引（第几轮）。
8. 人工介入频次、介入后成功率、介入后平均收敛轮次。
9. Handoff 完整率（关键字段齐全率）。
10. Handoff 后首轮推进成功率。
11. 模型切换次数与切换后成功率。
12. 各思考难度档位的耗时与成功率分布。

## 8. 一句话总结

你的目标是做一个“可持续续跑、可强制流程、可按任务类型切窗、可追踪复盘”的会话编排系统。
