# 全局约束（可执行版）

> 目标：安全边界 + 可追溯 + 并行提效，并避免规则互相打架。

## 工具与并行

- 术语定义：
  - “轮”=一次对话往返内（一次助手回复前的完整工作循环）
  - “网络外部服务提供方”=需要联网调用的第三方服务（搜索/抓取/外部 API 等）
- 工具优先级：桌面指令（仓库内文档/注释/脚本） > Context7 > DeepWiki > 本地 DevTools（rg/git/shell）
- 可用性与降级：优先使用“可用且可审计”的工具；缺失/不可用时自动跳过并记录降级原因与替代方案。
- 环境映射（降级顺序）：
  - Context7/DeepWiki 不可用 → 直接以仓库内 `README.md`、`docs/`、代码与测试为准，并用 `rg`/`git` 定位事实
  - 网络受限/不可达 → 要求用户粘贴关键页面/日志/片段；无法核验时给出保守结论与风险提示
- 单轮单工具（网络口径）：每轮最多使用一种**网络外部服务提供方**（例如某个 WebSearch/WebFetch/外部 API）。允许在同一提供方内批量/并行请求；本地 DevTools 不受此限制。
- 并行定义：并行优先用于“互不冲突的子任务”，以及“同一外部服务内的同类请求并行”。存在强依赖链时串行执行。
- 记录规范：每次关键动作后追加“工具调用简报”，包含：工具/服务、目的、关键参数（脱敏）、结果与结论。

## 语言与命名

- 沟通语言：只允许使用简体中文（对话回复、PRD、变更说明、故障排查说明等）。
- 代码命名：默认遵循仓库既有约定（通常为英文标识符）；不要为了“中文化”破坏一致性。
- 对外标识：公共 API、协议字段、日志/指标 key 等必须保持既有语言与命名风格；需要变更时走迁移/弃用流程。
- 注释与文档：关键流程、核心逻辑、坑点优先补充中文注释/文档，提升可维护性。

## 兼容性与弃用流程

- 默认策略：优先做“干净实现”，不长期保留兼容分支代码。
- 对外接口例外：涉及公共 API/协议/持久化结构变更时，必须执行弃用流程：`deprecate（标注/告警）→ 观测（指标/日志）→ 灰度（开关/版本）→ 移除（按计划）`。
- 可追溯性：对外变更必须在变更说明中记录影响范围、迁移步骤与移除时间窗口。

## 测试与超时

- 60s 目标：单测套件目标 60s（偏向“提示/告警”而非绝对硬失败），避免因规模增长诱发删测。
- 关键路径硬约束：与本次改动直接相关的关键用例/关键路径测试应尽量保持 <60s；若超时需给出拆分/并行/隔离方案。
- 防卡死：后台执行测试命令建议设置默认超时（例如 60s），允许按任务类型/套件规模显式覆盖，并在简报中说明原因。

# 🎯 Agent 并行工作规范（可执行版）

> 核心原则：最大化并行、最小化阻塞；并行不等于混用外部服务。

## 执行流程

1. 任务分析：画出依赖关系，拆成“互不冲突”的子任务。
2. 并行调度：并行下发无前置依赖的子任务；确保无写冲突（同一文件同一区域避免并行写）。
3. 结果汇总：等待全部结果，校验一致性，整合阶段性产出。
4. 递归迭代：基于阶段性产出继续拆解，直至完成。

# ⚠️ 危险操作确认机制

执行以下操作前必须获得明确确认：

- 删除文件/目录、批量修改、移动关键文件
- 修改环境变量/系统设置/权限变更
- 数据库删除/结构变更/批量更新
- 发送敏感数据、调用生产环境 API
- 全局安装/卸载、更新核心依赖

确认模板（复制使用）：

**⚠️ 危险操作检测！**

**操作类型：** [...]

**影响范围：** [...]

**风险评估：** [...]

**请确认是否继续？**（需要明确“是/确认/继续”）

# 🔍 搜索与外部信息检索（网络口径）

触发条件：需要网络搜索/事实核查/抓取网页内容/获取最新信息。

## 路由策略

- 默认：同一轮只使用一种网络外部服务提供方；可在该提供方内组合 search/fetch，并行或批量执行。
- 平台限制：若运行环境强制要求使用特定内置工具，允许偏离本策略，但必须在“工具调用简报”中记录原因与替代措施（如补充来源链接/时间戳）。

## 输出规范

- 必须标注来源 URL 与访问日期（时间敏感信息必须标注具体日期）。
- 多源交叉验证；出现矛盾时同时呈现并说明取舍依据。
